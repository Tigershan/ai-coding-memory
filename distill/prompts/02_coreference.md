# Prompt: 指代消解 + 解决方案提炼（Coreference Resolution + Solution Extraction）

> Phase 2 实现时直接读取此文件作为 prompt 模板。
> 完整设计见 `docs/design.md` 5.3 节。

---

你是对话改写与知识提炼专家。完成两个任务：

## 任务 1：指代消解

重写以下对话，使其完全自描述（任何后续读者无需上下文也能看懂）。

【消解规则】
- "这个模块" / "它" / "这块" → 替换为具体名称
- "上面那个方案" → 引用具体方案的关键词（如"基于 Redis 的限流方案"）
- "之前说的" → 找到具体内容补全
- "这里" → 替换为具体类名/方法名/文件路径

【用户纠正 AI 的标记规则】
当用户纠正了 AI 的错误回答时，这是极高价值的知识点。在重写时用 `⚡ CORRECTION` 标记突出：

标记方式：
- 在纠正发生的 user 消息前插入行：`> ⚡ CORRECTION: {一句话描述纠正了什么}`
- 保留用户的原始否定表达（"不对"、"错了"、"这个不行"），不要美化

特征信号（满足任一即标记）：
- 用户显式否定 AI 的输出（"不对"、"错了"、"不是这样"、"你搞混了"）
- 用户提供了 AI 不知道的项目事实来推翻 AI 的假设
- 用户指出 AI 方案在实际环境中不可行并给出正确做法
- AI 自己承认错误并修正（"抱歉，你说得对"、"我之前的回答有误"）

示例：
```
**AI**：建议使用 Guava RateLimiter 来实现限流...

> ⚡ CORRECTION: 分布式场景不能用 Guava RateLimiter，必须用 Redisson

**用户**：不对，我们是分布式部署，Guava RateLimiter 只支持单机。必须用 Redisson 的 RRateLimiter。

**AI**：你说得对，分布式场景下应该使用 Redisson RRateLimiter...
```

【保留】
- 保留原始问答的逻辑结构（user / assistant 交替）
- 保留用户的真实表达情绪（如"这个不行"、"我懵了"）
- 保留代码块原文（代码块的处理在下一步）

## 任务 2：解决方案提炼

如果对话中包含"探索 → 解决问题"的过程，在对话末尾追加一个结构化总结：

```
---
## 💡 解决方案摘要

**问题**：（一句话描述遇到的问题）
**根因**：（为什么会出现这个问题）
**解决方案**：（最终采用的方案，包含关键步骤）
**踩坑记录**：（尝试过但失败的方案，以及为什么失败）
```

如果对话不涉及问题解决（如纯知识学习、代码生成等），则跳过此部分，不要强行总结。

同样，如果对话中用户向 AI 传授了项目/环境特有知识（如"预发域名是 xxx"、"本地启动需要跳过 xx 插件"），在末尾追加：

```
---
## 📌 项目知识点

- （用户提供的知识点 1）
- （用户提供的知识点 2）
```

【输入】
workspace: {workspace}
topic_title: {topic_title}
topic_dialogue:
{topic_messages}

【输出格式（Markdown）】

**用户**：（重写后的提问，所有指代已消解）

**AI**：（重写后的回答，所有指代已消解）

**用户**：...

（如有解决方案 → 追加 ## 💡 解决方案摘要）
（如有项目知识 → 追加 ## 📌 项目知识点）

【约束】
- 不得新增信息（只能消解和提炼已有内容，不能编造）
- 若某个指代实在无法消解（上下文不足），保留原文并加 `[ref-unresolved]` 标记
- `⚡ CORRECTION` 标记仅在确实发生纠正时使用，不要对正常的追问或补充信息误标
- 输出末尾追加一行：`[coreference_confidence: 0.X]`
- 输出末尾追加一行：`[correction_count: N]`（N 为本 topic 中检测到的纠正次数，无纠正时为 0）
